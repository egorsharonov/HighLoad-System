# idealo — сервис сравнения цен и отслеживания скидок

## 1. Тема и целевая аудитория

### 1.1 Тема проекта
**idealo** — европейский сервис **сравнения цен** (price comparison): пользователь ищет товар, видит **единый профиль товара** и **предложения (офферы) магазинов** с ценой/доставкой/наличием, может отслеживать динамику цены и получать уведомления о снижении.

**MVP:** проектирование ключевой серверной части, связанной с поиском и выдачей офферов, историей цены и подписками пользователя.

### 1.2 Аналоги (для понимания класса сервиса)
Сервисы аналогичного типа: **PriceRunner**, **Google Shopping**, **Kelkoo**, **Geizhals** (ценовые агрегаторы / сравнение цен).

### 1.3 Целевая аудитория (размер и местоположение)
**Местоположение:** Европа (ядро — Германия), также присутствие в нескольких европейских странах.

**Размер аудитории / нагрузка (для обоснования highload):**
- idealo указывает **~78 млн посещений в месяц** (в среднем) и **~50 000 магазинов** (для Германии) — это порядок **≈2.6 млн посещений в день**.
- В материалах партнёров/кейсов также встречаются оценки **~72 млн пользователей в месяц** и «миллионы визитов в день», а также пиковые режимы уровня сотен тысяч запросов в секунду.

**Вывод:** требование курса **“≥1 млн пользователей/день”** по масштабу нагрузки выполняется с запасом.

> Примечание про “100+ серверов”: прямые цифры инфраструктуры в открытых источниках обычно не раскрываются. Для дальнейших этапов курсовой будем считать, что для сервиса с десятками миллионов посещений/месяц, сотнями миллионов офферов и пиковыми нагрузками требуется **порядок сотен вычислительных узлов** (включая контуры поиска/кэша/БД/стриминга), т.е. **100+ серверов** как реалистичный порядок величины.

### 1.4 MVP-функционал (6 ключевых функций)
1. **Поиск и каталог товаров:** строка поиска + базовые фильтры (категория/бренд/цена) и сортировки.
2. **Страница товара (канонический товар):** название/характеристики, агрегированная информация, ссылка на историю цены.
3. **Список офферов по товару:** предложения магазинов (цена, доставка, наличие) + сортировка по “итоговой цене”.
4. **История цены товара:** график/минимум/максимум/динамика за период (на основе тайм-серий цен).
5. **Избранное (watchlist):** добавление/удаление товара, список отслеживаемых товаров пользователя.
6. **Уведомления о снижении цены:** правило “уведомить, если цена ≤ X” (или “упала на Y%”) + настройки частоты уведомлений.

### 1.5 Ключевые продуктовые решения (важные для архитектуры)
- **Разделение сущностей “Товар” и “Оффер”:**
  - *Товар* — нормализованная карточка (бренд/модель/атрибуты).
  - *Оффер* — предложение конкретного магазина (цена/доставка/наличие/ссылка/время обновления).
- **Поиск как отдельный контур:** для быстрого поиска/фильтров используется поисковый индекс (инвертированный индекс + фасеты).
- **Хранение истории цен:** цены — это тайм-серии; для графиков нужны агрегации (по дням/неделям) и быстрые выборки по товару.
- **Уведомления как событийная обработка:** обновление цены → проверка подписок → постановка задач на отправку уведомлений (с антиспам-лимитами).
- **Кэширование “горячих” запросов:** топ-товары и популярные страницы товара/офферы кэшируются для устойчивости в пиковые периоды.
